## Spacing Scale

Declared values (multiples of 4 only). Reused from existing phases — do not override.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, chip padding |
| sm | 8px | Item row spacing, inline gaps |
| md | 16px | Card padding, section padding |
| lg | 24px | Section breaks, list content padding |
| xl | 32px | Empty state vertical padding |
| 2xl | 48px | Page-level vertical spacing |
| 3xl | 64px | Email template outer padding |

Exceptions:
- Touch targets (Android): minimum 48dp height on all tappable rows and buttons — applies to "I completed the purchase" button and notification settings toggle row.
- Web confirm-purchase button: minimum height 48px for mobile touch parity.
- Email CTA button: 44px height minimum (table-based bulletproof button; not touch-target enforced by OS).

## Component Contracts

### 1. Confirm-Purchase Banner / Card

**Surfaces:** Android RegistryDetailScreen (sticky, above item list) + web RegistryPage (sticky, below reservation countdown banner).

**Trigger condition:** giver has an active reservation (`status: "active"`) — shown only to the device/tab that created the reservation, using the same `useActiveReservation` state (web) and `GuestPreferencesDataStore` (Android) from Phases 4–5.

**Android layout:**
- Surface: `Card` with `containerColor = MaterialTheme.colorScheme.surfaceVariant`, `elevation = 2.dp`
- Positioned as a sticky header item inside the `LazyColumn`, directly below the existing `ReservationCountdown` display
- Height: minimum 64dp
- Left icon: `Icons.Default.ShoppingCart` (24dp), tint `MaterialTheme.colorScheme.primary`
- Body text (labelLarge, 14sp, medium): `"Did you complete the purchase?"` / `"Ai finalizat achiziția?"`
- CTA button: `Button` (filled, primary), full width, height 48dp, labelLarge weight
  - Label: `"I completed the purchase"` / `"Am finalizat achiziția"`
  - Loading state: `CircularProgressIndicator` (size 20dp) inside button, button disabled
- No dismiss button. Card auto-disappears when `status` transitions to `purchased` or `released`.

**Web layout:**
- `div` with `role="status"` and `aria-live="polite"`
- Positioned sticky below existing `ReservationBanner` (top offset: top-24 if banner is visible, top-12 if not — banner is `sticky top-12 h-12`)
- Background: `bg-surface-variant` (#E7E0EC), text: `text-surface-on` (#1C1B1F)
- Height: minimum 64px, padding: px-4 py-3
- Left icon: shopping-cart SVG (20px), color: `text-primary`
- Body text: `text-sm font-bold` — `"Did you complete the purchase?"`
- CTA button: `bg-primary text-primary-on rounded-md px-4 py-2 text-sm font-bold min-h-[48px]`
  - Label: `"I completed the purchase"` / `"Am finalizat achiziția"` (i18next key: `reservation.confirm_purchase_cta`)
  - Loading state: spinner replaces label text; button disabled + `opacity-60`
- No dismiss button. Component unmounts when `active` reservation clears from state.

**Accessibility:** `aria-busy="true"` on button during loading. Button has visible focus ring (ring-2 ring-primary ring-offset-2).

### 2. Confirm-Purchase Success Feedback

**Android:** After `confirmPurchase` callable returns success:
- Snackbar message (existing `SnackbarHostState`): `"Purchase confirmed. Thank you!"` / `"Achiziție confirmată. Mulțumim!"`
- Snackbar duration: `SnackbarDuration.Short`
- Item status chip on the confirmed item transitions to `PURCHASED` via Firestore real-time listener (no extra UI needed)

**Web:** After `confirmPurchase` callable returns success:
- `showToast("Purchase confirmed!", "success")` via existing `ToastProvider` / Radix Toast
- `useActiveReservation().clear()` — banner unmounts
- Item card status chip updates to "Purchased" via existing onSnapshot listener

### 3. Foreground FCM Notification (Android)

**Context:** Owner's app is open (foreground). FCM notification arrives for a gift purchase.

**Pattern:** Use existing `SnackbarHostState` in the active screen (RegistryDetailScreen or RegistryListScreen). The `FirebaseMessagingService` broadcasts a local event (via a SharedFlow in a singleton `NotificationEventBus`) that the active screen collects.

**Visual:**
- Snackbar, duration `SnackbarDuration.Long` (10 seconds)
- Message: `"A gift was purchased from {registryName}!"` (i18n key: `notifications_purchase_snackbar`)
- Action label: `"View"` (i18n key: `notifications_purchase_snackbar_action`) — space-constrained Material3 Snackbar exception; single-word action label is intentional and consistent with M3 Snackbar guidelines for constrained action areas. Navigates to the registry detail screen for `data.registryId`.
- No custom drawable; standard M3 Snackbar

**Background FCM:** OS-rendered system notification. No custom notification layout in v1. Notification channel: `purchase_notifications`, importance: `IMPORTANCE_DEFAULT`.

**Notification channel metadata (Android):**
- Channel ID: `purchase_notifications`
- Channel name (string resource): `notifications_channel_purchase_name` → `"Purchase alerts"`
- Channel description: `notifications_channel_purchase_description` → `"Notified when a gift is purchased from your registry"`
- Importance: `NotificationManager.IMPORTANCE_DEFAULT`

### 4. Notification Settings Row (Owner — Android)

**Location:** Registry create/edit screen (existing). REG-09 toggle `notifyOnPurchase` already implemented in Phase 3.

**No new UI required.** The existing `Switch` + label row (`registry_notifications_label`, `registry_notifications_enabled`) is the correct surface. This phase does not move or modify it.

**Confirm:** The settings row is already wired to `notifyOnPurchase` field. Phase 6 only adds the server-side read of this field in the notification trigger — zero Android settings UI change.

---

### 5. Owner Preferred Locale Setting (Android + Web)

**D-14:** `users/{uid}.preferredLocale` is read server-side to pick en/ro email/notification template.

**Android:** Add a new `ListItem` row to `SettingsScreen` below the existing language row:
- Headline: `"Email language"` / `"Limba emailurilor"` (key: `settings_email_language_label`)
- Supporting text: current value — `"English"` or `"Romanian"`
- Leading icon: `Icons.Default.Email` (24dp)
- On tap: same `AlertDialog` pattern as existing language picker (RadioButton list with English / Romanian options)
- Default: seeded from `LanguagePreferencesDataStore` current locale at registration time, stored as `preferredLocale` field; subsequent changes via this picker write only `preferredLocale` (not app display language)

**Web (LanguageSwitcher):** The existing `LanguageSwitcher` component controls display language only. Phase 6 does NOT add a separate email-language control to the web fallback. The web fallback is giver-only; givers do not receive purchase-notification emails. The web-fallback owner settings UI scope is deferred (owners use the Android app for settings).

### 6. Email Templates (New Surface)

Three templates: `invite`, `expiry`, `purchase`. All share a common layout wrapper.

#### Common Email Shell

```
[Header band — full width, 64px, bg #6750A4]
  [Wordmark: "Gift Registry" — Arial bold 20px, color #FFFFFF, centered]

[Body — full width, bg #FFFBFE]
  [Content card — 600px max-width, centered, bg #FFFFFF, padding 32px]
    {template content}

[Footer — 600px max-width, centered]
  [Divider — 1px solid #CAC4D0]
  [Legal/support text — Arial 12px, color #49454F, centered]
  "You received this email because you used Gift Registry."
  "Romanian variant: Ai primit acest email deoarece ai folosit Gift Registry."
```

#### CTA Button (bulletproof — table-based)

```html
<table cellpadding="0" cellspacing="0" border="0" style="margin: 24px auto;">
  <tr>
    <td align="center" bgcolor="#6750A4"
        style="border-radius:6px; padding:0;">
      <a href="{{ctaUrl}}"
         style="display:inline-block; padding:14px 32px;
                font-family:Arial,Helvetica,sans-serif;
                font-size:16px; font-weight:bold;
                color:#FFFFFF; text-decoration:none;
                border-radius:6px;">
        {{ctaLabel}}
      </a>
    </td>
  </tr>
</table>
```

Height: ~44px (padding 14px top + 14px bottom + 16px font). No `min-height` — email clients do not support it reliably.

## Copywriting Contract

| Element | Copy (en) | Copy (ro) | i18n key |
|---------|-----------|-----------|----------|
| Confirm-purchase CTA | I completed the purchase | Am finalizat achiziția | `reservation.confirm_purchase_cta` |
| Confirm-purchase card heading | Did you complete the purchase? | Ai finalizat achiziția? | `reservation.confirm_purchase_heading` |
| Confirm-purchase loading | Confirming… | Se confirmă… | `reservation.confirm_purchase_loading` |
| Confirm-purchase success toast | Purchase confirmed. Thank you! | Achiziție confirmată. Mulțumim! | `reservation.confirm_purchase_success` |
| Confirm-purchase error toast | Could not confirm. Please try again. | Nu s-a putut confirma. Încearcă din nou. | `reservation.confirm_purchase_error` |
| Foreground notif snackbar | A gift was purchased from {{registryName}}! | Un cadou din {{registryName}} a fost cumpărat! | `notifications_purchase_snackbar` |
| Notif snackbar action | View | Vezi | `notifications_purchase_snackbar_action` |
| Email language settings label | Email language | Limba emailurilor | `settings_email_language_label` |
| Email language dialog title | Email language | Limba emailurilor | (same key) |
| Notification channel name | Purchase alerts | (Android only — not localized at channel creation time; use English) | `notifications_channel_purchase_name` |
| Notification channel description | Notified when a gift is purchased from your registry | (Android only — English) | `notifications_channel_purchase_description` |

Note on `notifications_purchase_snackbar_action` (`"View"`): single-word label is a justified space-constrained Material3 Snackbar exception. M3 Snackbar action areas are narrow; single imperative verbs are the standard pattern in this context.

**Empty states:** No new empty states introduced in this phase. Existing reservation empty state (`reservation.expired`) handles the post-expiry case.

**Error states:**
- Confirm-purchase network error: `"Could not confirm. Please try again."` — displayed as error toast (web) / Snackbar (Android).
- FCM token registration failure: silent — no user-facing message (server-logged per D-15).
- Email delivery failure: silent — no user-facing message (logged to `notifications_failures` per D-15).

**Destructive actions in this phase:** None. Confirm-purchase is a positive action, not destructive. No confirmation dialog required.
