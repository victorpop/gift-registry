<objective>
Gate Firebase emulator wiring behind a Gradle project property `use_emulator` so on-device debug builds can target production Firebase without source changes, while keeping the default dev loop (emulator-backed debug build) unchanged.

Purpose: Unblock real-device debug testing. Today `AppModule.kt` unconditionally calls `useEmulator("10.0.2.2", ...)` when `BuildConfig.DEBUG` is true, which fails on physical phones because `10.0.2.2` only resolves inside the Android emulator (it is the host alias for the emulator's loopback).

Output: Two-file change — `app/build.gradle.kts` grows a `buildTypes { debug { ... } release { ... } }` block that emits a boolean `BuildConfig.USE_FIREBASE_EMULATOR` field; `AppModule.kt` flips three gate sites from `BuildConfig.DEBUG` to `BuildConfig.USE_FIREBASE_EMULATOR`. Release builds always get `false` hardcoded.
</objective>

```kotlin
        // Opt out for on-device testing with: ./gradlew :app:assembleDebug -Puse_emulator=false
        val useEmulator = providers.gradleProperty("use_emulator").getOrElse("true")

```

Critical correctness notes:
- The third argument to `buildConfigField` is the source-code expression that will be inlined into `BuildConfig.java`. For a `boolean` field the string must be literally `"true"` or `"false"` — NOT `"\"true\""`. Wrapping in extra quotes would produce a `String` field, which breaks the `if (BuildConfig.USE_FIREBASE_EMULATOR)` check in Kotlin.
- Do NOT introduce a new product flavor — buildConfigField on buildTypes is the idiomatic path.
- Do NOT touch `buildFeatures { buildConfig = true }` — it is already enabled, which is why BuildConfig generation works.
- Do NOT edit `gradle.properties` to set a default — keeping the property unset lets `.getOrElse("true")` own the default, which keeps the dev loop unchanged for anyone who pulls without reconfiguring.

Do not modify any other section of the file.
